----------------------------------------------------------------------
OVERVIEW
----------------------------------------------------------------------
CrateMindAI is a local-first, pipeline-based DJ library automation toolkit.

It transforms messy audio downloads into a clean, structured,
Rekordbox-ready library using a layered approach:

  1. Deterministic cleaning   — offline, rule-based, no dependencies
  2. AI normalization         — local Ollama LLM, never sends data externally
  3. Artist identity          — alias resolution, canonical names
  4. Online enrichment        — Spotify + Deezer, strict confidence gates
  5. Label intelligence       — label parsing + enrichment (evolving)

Every stage is standalone. Run one step, or compose the full pipeline.
Nothing writes without --apply. Every change is logged.


----------------------------------------------------------------------
PIPELINE STAGES
----------------------------------------------------------------------

Stage 1: metadata-sanitize
  Deterministic, offline, no AI.
  Removes URL watermarks, promo artifacts, DJ pool tags, malformed ISRCs.
  Safe to run before anything else. Idempotent.

Stage 2: ai-normalize
  Local Ollama LLM proposes improvements to:
    artist, title, version, label, remixers, featured_artists
  Minimum confidence: 0.75. BPM / key / cues: NEVER touched.
  --pre-sanitize: runs stage 1 before inference (recommended).

Stage 3: artist-intelligence
  Deterministic artist normalization:
    - Normalizes to canonical form
    - Detects and stores aliases
    - Resolves collab/feat variants
  Package: intelligence/artist/

Stage 4: metadata-enrich-online
  Fills missing album / label / ISRC via external APIs.
  Source priority: Spotify ISRC → Spotify artist+title → Deezer → Traxsource
  Minimum confidence: 0.80.
  Package: intelligence/enrichment/

Stage 5: label-intelligence
  Label parsing and enrichment (intelligence/label/).
  Still evolving.


----------------------------------------------------------------------
OPERATIONAL STATES (metadata-enrich-online)
----------------------------------------------------------------------
Each track receives one of three outcomes per enrichment run:

  APPLY    conf >= 0.80; all safety rules pass
           Written to file with --apply.

  REVIEW   0.70 <= conf < 0.80; no hard block
           Added to review queue: data/intelligence/enrichment_review_queue.json
           Resolved interactively via: python3 pipeline.py review-queue

  SKIP     Hard safety block fires (version mismatch, low artist similarity)
           Not applied. Moved to IGNORED quarantine with --move-ignored.
           IGNORED path: /home/koolkatdj/Music/music/IGNORED/


----------------------------------------------------------------------
SAFETY MODEL
----------------------------------------------------------------------
Hard blocks — always enforced regardless of confidence:
  - artist field: NEVER proposed or modified
  - Version mismatch: conflicting version tokens cap confidence at 0.74
  - Low artist similarity (< 0.90 without ISRC anchor): cap at 0.74
  - ISRC exact match overrides formula (confidence 0.98)

Change policy:
  title     only if current is blank, or ISRC matched + base differs
  album     if current is empty, or ISRC matched
  label     if current is empty; or ISRC matched + conf >= 0.95
  isrc      only if not already set

MIK-first policy (non-negotiable):
  BPM, key, and cue data are NEVER modified.
  Mixed In Key is the authoritative source.
  Pipeline only fills gaps left by MIK.


----------------------------------------------------------------------
IGNORED QUARANTINE
----------------------------------------------------------------------
Path: /home/koolkatdj/Music/music/IGNORED/

Populated by: metadata-enrich-online --move-ignored
Triggered by: decision_code in {skipped_low_score, skipped_artist_mismatch,
                                  skipped_version_conflict}

Behavior:
  - Relative folder structure is preserved under IGNORED/
  - Collision handling: appends _dup1, _dup2, … to filename
  - Off by default — only active when --move-ignored is explicitly passed
  - Threshold skips (conf below threshold but no hard block) are NOT moved


----------------------------------------------------------------------
IDEMPOTENCY
----------------------------------------------------------------------
The pipeline is safe to re-run at any stage:
  - Already-processed tracks (TXXX:PROCESSED=1 + DB status=ok) skipped
  - Review queue deduplicates by file_path (repeat runs refresh entries)
  - metadata-sanitize on a clean file produces no changes
  - analyze-missing does not overwrite valid BPM/key values

----------------------------------------------------------------------
DATABASE SCHEMA
----------------------------------------------------------------------
SQLite (processed.db):
  tracks           filepath, artist, title, album, genre, bpm, key_camelot,
                   key_musical, duration_sec, bitrate_kbps, quality_tier,
                   status: pending|ok|rejected|duplicate|needs_review|error
  track_history    rollback snapshots (original_meta, cleaned_meta, actions)
  pipeline_runs    audit trail per run
  duplicate_groups original, duplicate, reason, resolved
  cue_points       suggested cue positions

SQLite (jobs.db — backend only):
  jobs             id, status, command, pid, progress fields
  bpm_anomalies    filepath, reason, review_status


----------------------------------------------------------------------
WEB APP
----------------------------------------------------------------------
The web app provides a UI for:
  - Job management (trigger + monitor CLI subcommands)
  - Library browsing (tracks, issues, stats)
  - BPM anomaly review
  - Set builder
  - Export + SSD sync

IMPORTANT: The Collection workspace is NOT the source of truth.
The CLI pipeline remains authoritative. The web app reads from the
same SQLite database that the CLI writes.
