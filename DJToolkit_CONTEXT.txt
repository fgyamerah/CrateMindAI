----------------------------------------------------------------------
PURPOSE
----------------------------------------------------------------------
CrateMindAI is a local-first, pipeline-based DJ library automation toolkit.

Transforms raw audio downloads into:
- Clean, structured metadata
- BPM / key (MIK-compliant)
- Organized library
- Rekordbox-compatible exports

Platform: Ubuntu Studio 24 → Windows-compatible DJ drive output.

Design principles:
- Local-first: no mandatory cloud dependency
- Deterministic before AI: rules run first, AI fills gaps
- AI before enrichment: normalize identity before online lookup
- Enrichment only when identity is proven: no guessing
- Preview by default: nothing writes without --apply
- Idempotent: safe to re-run at any point in the pipeline
- MIK-first: Mixed In Key is the authoritative source for BPM, key, and cues


----------------------------------------------------------------------
INTELLIGENCE PIPELINE (primary workflow)
----------------------------------------------------------------------
Pipeline order (each stage is standalone; compose as needed):

  1. metadata-sanitize      Deterministic offline cleaning
  2. ai-normalize           Local Ollama AI proposals
  3. artist-intelligence    Canonical artist normalization
  4. metadata-enrich-online Spotify + Deezer metadata enrichment
  5. label-intelligence     Label parsing + enrichment (evolving)

Each stage: preview by default, --apply to write.

----------------------------------------------------------------------
MIK-FIRST POLICY (NON-NEGOTIABLE)
----------------------------------------------------------------------
Mixed In Key is the authoritative source for BPM, key, and cue data.

NEVER:
- Overwrite existing BPM
- Overwrite existing key
- Overwrite cue points
- Re-analyze if data already exists

ALWAYS:
1. Check DB and file tags first (_read_existing_analysis())
2. Preserve existing values
3. Only fill missing data

XML:
- Disabled by default
- Only use --force-xml if MIK is not used

M3U:
- Always safe


----------------------------------------------------------------------
ENRICHMENT SAFETY MODEL (metadata-enrich-online)
----------------------------------------------------------------------
Source priority:
  1. Spotify ISRC       (most precise; skipped if no ISRC on file)
  2. Spotify artist+title
  3. Deezer             (fallback when Spotify conf < 0.70)
  4. Traxsource         (dance-music specialist; when best < 0.80 or genre
                         matches house/afro/deep/amapiano etc.)

Operational states per track:
  APPLY   conf >= 0.80; all safety rules pass -> written with --apply
  REVIEW  0.70 <= conf < 0.80 -> added to review-queue
  SKIP    hard safety block fires -> moved to IGNORED with --move-ignored

Hard blocks (always enforced):
  - artist field: NEVER proposed
  - Version mismatch: conflicting version tokens -> cap at 0.74
  - Low artist similarity (< 0.90, no ISRC): cap at 0.74
  - ISRC exact match: confidence 0.98, bypasses formula limits

Change policy:
  title     : only if current is blank, or ISRC matched + base differs
  album     : if current is empty, or ISRC matched
  label     : if current is empty; or ISRC matched + conf >= 0.95
  isrc      : only if not already set

IGNORED quarantine path: /home/koolkatdj/Music/music/IGNORED/
  Preserves relative folder structure. Collision: appends _dup1, _dup2 …
  Triggered by --move-ignored flag only (off by default).


----------------------------------------------------------------------
REVIEW QUEUE
----------------------------------------------------------------------
CLI: python3 pipeline.py review-queue [--list-only]

Queue file: data/intelligence/enrichment_review_queue.json
Populated by metadata-enrich-online for REVIEW-state results.
Deduplicated by file_path — repeat runs refresh entries, not grow queue.

Interactive actions:
  a = apply proposed changes + remove entry
  s = skip (keep in queue)
  d = delete entry without applying
  n = next
  q = quit


----------------------------------------------------------------------
AI NORMALIZE (ai/)
----------------------------------------------------------------------
Subcommand: ai-normalize
Local Ollama LLM proposals for: artist, title, version, label,
remixers, featured_artists.

Preview by default; --apply to write.
Min confidence: 0.75 (--min-confidence flag).
--pre-sanitize: runs metadata-sanitize before inference (recommended).

Dataset logging (data/intelligence/):
  review_queue.jsonl, accepted_examples.jsonl, rejected_examples.jsonl,
  review_decisions.jsonl, fewshot_examples.jsonl

Config keys:
  OLLAMA_BASE_URL, OLLAMA_DEFAULT_MODEL, OLLAMA_TIMEOUT


----------------------------------------------------------------------
ARTIST INTELLIGENCE (intelligence/artist/)
----------------------------------------------------------------------
Subcommand: artist-intelligence
Deterministic artist normalization + alias store.

Three-category merge classification (also used by artist-merge):
  SAFE_ALIAS           Formatting/casing/punctuation difference only
  SAME_PRIMARY_COLLAB  Primary matches; >=1 folder has feat/ft suffix
  AMBIGUOUS            Primary artists differ after normalization -> never merged

Canonical name selection: most files wins. Tiebreakers: penalize
all-uppercase, all-lowercase; then alphabetical.
