[2026-04-15] — Add ai-normalize subcommand (local Ollama metadata proposals)
----------------------------------------------------------------------
TYPE: New feature
SUMMARY: Adds `python3 pipeline.py ai-normalize` — a safe, preview-first
  workflow that uses a local Ollama LLM to propose normalized artist/title/
  version/label/remixers/featured_artists metadata for a batch of audio files.

DETAILS:
  - New ai/ package: ollama_client.py (urllib, no new deps), metadata_schema.py
    (strict validated dataclass), normalizer.py (prompt builder, JSON extractor,
    diff engine, apply step, CLI entry point)
  - OllamaClient uses stdlib urllib.request; healthcheck + generate endpoints
  - NormalizedMetadata.from_dict() tolerates malformed model output (wrong types,
    sentinel strings, confidence out of range) — never raises
  - _extract_json() handles plain JSON, markdown code fences, and mixed text
  - compute_diff() only surfaces fields where the proposal is non-empty AND
    different from the current value — no spurious diffs
  - apply_normalized() writes using mutagen easy tags (same pattern as
    sanitizer._write_tags); artist + featured_artists combined to "A feat. B";
    title + version combined to "Track (Original Mix)"
  - Reuses parse_filename_stem() for deterministic pre-AI context
  - Reuses sanitize_text() to strip residual junk from model output
  - Reuses config.AUDIO_EXTENSIONS + same rglob dedup pattern as _collect_inbox()
  - Safe by design: preview is the default; --apply is required to write anything;
    --min-confidence gate (default 0.75); BPM/key/cue never touched
  - config.py gains OLLAMA_BASE_URL, OLLAMA_DEFAULT_MODEL, OLLAMA_TIMEOUT
    (all overridable via env vars or config_local.py)

FILES:
  ai/__init__.py          (new)
  ai/ollama_client.py     (new)
  ai/metadata_schema.py   (new)
  ai/normalizer.py        (new)
  config.py               (added OLLAMA_* constants)
  pipeline.py             (ai-normalize subparser + lazy dispatch import)

IMPACT: Additive only. No existing module changed structurally. The lazy import
  (`from ai.normalizer import run_ai_normalize`) means the ai/ package is never
  loaded unless the user explicitly runs the ai-normalize subcommand, so there is
  zero startup-time cost for all other commands.

NOTES:
  - Default model: qwen2.5-coder:3b (already installed per user setup)
  - Ollama must be running: `ollama serve`
  - No cloud calls; no API keys; fully local inference
  - v1 scope: artist, title, version, label, remixers, featured_artists only
    (no BPM, key, genre, energy — those are MIK-owned or handled elsewhere)

[2026-04-12] v0.6.0 — SSD Sync, job cancellation, and progress tracking
----------------------------------------------------------------------

TYPE: feature

SUMMARY:
Adds SSD Sync page (rsync library → SSD target), cancellation for all running
jobs via SIGTERM, and progress tracking (current/total/percent/message) stored
in the jobs DB and displayed as progress bars in the UI.

DETAILS:
Backend — new files:
  backend/app/services/process_registry.py
      Shared in-process table of running asyncio subprocesses (job_id →
      asyncio.subprocess.Process).  Both toolkit_runner and rsync_runner
      register here so the cancel route works for all job types.
      request_cancel() adds job to _cancelling set and sends SIGTERM.
      Background tasks check is_cancelling() to choose 'cancelled' vs 'failed'.

  backend/app/schemas/sync.py
      SyncPreviewRequest, SyncPreviewResponse, SyncFileChange,
      SyncRunRequest, SyncRunResponse, SyncConfigResponse

  backend/app/services/rsync_runner.py
      preview_sync() — async dry-run rsync, parses file list + stats,
        returns structured SyncPreviewResponse (no DB writes, no job).
      start_sync_job() — validates paths, creates job record, launches
        _run_rsync_job() as asyncio background task.
      _run_rsync_job() — reads rsync stdout line-by-line, writes to log
        file, parses --info=progress2 lines for to-chk=N/M progress,
        throttled to one DB write per second.
      _parse_progress_line() — regex parse for to-chk=remaining/total.
      get_sync_config() — returns configured paths + SSD mount status.
      Path validation: source must be 'library' or 'inbox' key (never

      GET  /api/sync              — list past ssd-sync jobs
      GET  /api/sync/{job_id}     — get single sync job

----------------------------------------------------------------------

TYPE: feature

SUMMARY:
Adds a BPM anomaly detection pipeline: scans existing BPM values in the
library DB using heuristic rules (same logic as analyzer.py corrections),
persists anomaly records with human review state in the backend DB, and
surfaces them in a dedicated BPM Review page with per-row actions and

DETAILS:

  backend/app/services/bpm_analysis.py — classify_bpm() logic + scan/upsert/list/update

                                        UpdateAnomalyRequest, ReanalyzeRequest
  backend/app/api/routes/analysis.py  — POST /bpm-check, GET /bpm-anomalies,
                                        GET /bpm-anomalies/summary, PATCH /bpm-anomalies/{id},
                                        POST /reanalyze
  backend/app/main.py                 — registered analysis_router

Frontend:
  frontend/src/types/analysis.ts      — BpmAnomaly, BpmCheckResult, BpmSummary types
  frontend/src/api/analysis.ts        — all analysis API calls
  frontend/src/api/client.ts          — added apiFetch.patch()
  frontend/src/pages/BpmReview.tsx    — BPM Review page with scan, summary, filter table,
                                        per-row actions, inline job log viewer
  frontend/src/App.tsx                — added /bpm-review route
  frontend/src/components/Sidebar.tsx — added BPM Review nav link
  frontend/src/index.css              — reason badges, anomaly status badges, summary bar,
                                        reanalyze panel, job log viewer, anomaly table styles

BPM LOGIC (classify_bpm):
  missing_bpm    : NULL or zero stored
  too_low_10x    : 0 < BPM < 20  → suggest × 10  (e.g. 12.1 → 121)
  likely_halved  : 20 ≤ BPM < 90 → suggest × 2   (e.g. 64 → 128)
  likely_doubled : 160 < BPM ≤ 240, not DNB/jungle/hardcore → suggest ÷ 2
  too_high       : BPM > 240 → suggest ÷ 2

WHY:
  Incorrect BPM (especially doubled/halved) is a common aubio artifact.
  The analyzer already corrects these at analysis time, but values that
  came from MIK, Rekordbox, or direct tag writes bypass that correction.
  This feature surfaces those cases without touching files.

NOTES:
  - bpm_anomalies table is in backend jobs.db, NOT pipeline processed.db
  - Human review state (reviewed/ignored/requeued) is preserved across re-scans
  - reanalyze endpoint uses existing analyze-missing + --reanalyze via job system

[2026-04-12] v1.7.0 — Artist merge: three-category classification, normalization fix, tests
----------------------------------------------------------------------

TYPE: feature / bug fix

SUMMARY:
Refactored artist-merge classification so that punctuation/casing/hyphen variants
(e.g. "Heavy-K" / "Heavy K", "H.O.S.H" / "HOSH") are correctly recognised as
safe aliases instead of being marked ambiguous. Introduced three explicit merge
categories and 77 new tests.

DETAILS:
Root cause of the bug:
  The old _classify_merge() compared fi.primary_artist.lower() directly.
  "Heavy-K".lower() = "heavy-k" ≠ "heavy k" = "Heavy K".lower(), so despite
  being grouped correctly, the comparison flagged them as different primaries.

Fix:
  New _normalize_primary_for_compare() applies the same transforms used by
  normalize_artist_key() (period strip, hyphen→space, etc.) before comparison.
  Now "Heavy-K" and "Heavy K" both normalize to "heavy k" → SAFE_ALIAS.

Three merge categories (replacing the old binary safe/unsafe):
  SAFE_ALIAS          — formatting/punctuation/casing differences only; no collab.
                        Examples: "Heavy-K" / "Heavy K", "H.O.S.H" / "HOSH",
                        "Rosalie" / "Rosalie.", "culoe de song" / "Culoe De Song"
  SAME_PRIMARY_COLLAB — same primary after normalization; ≥1 folder has feat/comma.
                        Examples: "Heavy-K" / "Heavy-K feat. Davido"
  AMBIGUOUS           — primary artists differ after normalization; never auto-merged.

Other improvements:
  - _describe_alias_differences() generates human-readable reason strings:
      "capitalization variant", "dotted-initials variant", "hyphen/space variant",
      "trailing period variant", "period variant", "underscore/space variant",
      "quotation style variant"
  - _pick_canonical() penalises all-uppercase names (e.g. HOSH loses to H.O.S.H)
    and all-lowercase names; mixed-case preferred.
  - FolderInfo.has_collab_suffix field added (populated at scan time).
  - MergeGroup.merge_category field added.
  - build_merge_groups() returns (safe, ambiguous) where safe = SAFE_ALIAS +
    SAME_PRIMARY_COLLAB groups; ambiguous = AMBIGUOUS only.
  - JSON report restructured with safe_alias_merges, same_primary_collab_merges,
    ambiguous_merges sections.
  - Console output shows all three categories with clear labels.
  - New constants: MERGE_CATEGORY_SAFE_ALIAS, MERGE_CATEGORY_SAME_PRIMARY_COLLAB,
    MERGE_CATEGORY_AMBIGUOUS.

FILES:
  modules/artist_merge.py     — full refactor (normalization fix, new functions,
                                 updated dataclasses, report structure)
  tests/test_artist_merge.py  — new: 77 tests across 7 test classes

IMPACT:
  - Behaviour change: groups that were previously marked AMBIGUOUS (and skipped)
    because of hyphen/period/case differences are now correctly classified as
    SAFE_ALIAS and auto-applied in --apply mode.
  - No change to file-move logic, DB updates, or other pipeline steps.

[2026-04-11] v1.3.0 — Sanitizer: strip leading track-index junk before pipe
----------------------------------------------------------------------

TYPE: feature / patch

SUMMARY:
- New regex rule in sanitize_text() removes junk prefixes (digit, single letter,
  or symbol+digit/letter) that appear at the very start of a title before a `|`
  character and a space.  Handles DJ-pool numbering like "2 | Heartbreaker" or
  "***3 | Track (Remix)" and single-letter prefixes like "A | Charlotte".

DETAILS:
- Pattern: `^[^\w\s]*(?:\d+|[A-Za-z])\s*\|\s*`
  - `^`         — anchored to start; mid-string pipes are NEVER touched
  - `[^\w\s]*`  — zero or more leading symbol chars (e.g. `***`)
  - `(?:\d+|[A-Za-z])` — one or more digits OR exactly one letter (single-char only)
  - `\s*\|\s*`  — optional spaces around the pipe (all consumed)
- Multi-letter words before a pipe (e.g. "Rock | Roll", "Day | Night") are
  preserved because the single `[A-Za-z]` token requires the next char to be
  the pipe, not another word character.
- Runs as Step 0 in sanitize_text(), before URL and promo-phrase passes.
- 8 new test cases added to TestSanitizeText.

FILES:
- modules/sanitizer.py     — added _RE_PIPE_JUNK_PREFIX regex + Step 0 in sanitize_text()
- tests/test_sanitizer.py  — 8 new tests for the pipe-prefix rule

IMPACT:
- Any title/field string starting with a short numeric or single-letter index
  before a `|` will now be cleaned automatically in the sanitize pass.
- No impact on MIK BPM/key/cue data (sanitizer never touches those fields).

- This is additive; all 76 existing tests pass.
