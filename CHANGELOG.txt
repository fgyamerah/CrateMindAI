[2026-04-19] — ai-normalize: add --pre-sanitize flag to run metadata-sanitize first
----------------------------------------------------------------------
TYPE: New flag (additive, no behavior change when absent)
SUMMARY: Adds --pre-sanitize to the ai-normalize subcommand. When present, runs
         metadata-sanitize --apply on the input path before AI inference begins,
         so the model receives clean tags rather than junk-polluted input.

DETAILS:
- New argument registered in the ai-normalize subparser in pipeline.py
- Dispatch block checks getattr(args, "pre_sanitize", False) before calling
  run_ai_normalize; if True, constructs a sanitize Namespace and calls
  run_metadata_sanitize() first
- Sanitize Namespace forwards: input, apply, limit, verbose from the parent args;
  output_json is always None (sanitize JSON log not needed in this mode)
- If run_metadata_sanitize() exits non-zero, the dispatch aborts before ai-normalize
  runs — partial sanitize failure never feeds a broken state into the model
- --apply is the single write gate for both steps:
    no --apply → sanitize previews, ai-normalize previews
    --apply    → sanitize writes to disk, ai-normalize reads clean tags and writes
- Without --pre-sanitize, the ai-normalize dispatch path is byte-for-byte identical
  to before — zero regression risk
- modules/metadata_sanitize.py and ai/normalizer.py are not modified

[2026-04-18] — New metadata-sanitize command: offline, deterministic tag sanitation
----------------------------------------------------------------------
TYPE: New feature / new CLI command
SUMMARY: Adds a standalone metadata-sanitize subcommand that cleans tags before any
         enrichment or AI layer runs. Fully offline, no network calls, no AI.

DETAILS:
- Covers 5 fields: album, isrc, title, artist, organization (label)
- album:  clears if value contains a URL, path fragment, 3+ consecutive dots, or promo junk
          (delegates to existing sanitize_text() for URL/domain/promo phrase detection)
- isrc:   clears if value does not match strict ISRC format (CC-XXX-YY-NNNNNNN or 12-char bare)

- title:  strips leading numeric prefixes (e.g. "01. Track", "1 | Track") when an explicit
          separator is present (safe — "01 Luftballons" is NOT stripped)
          Also fixes duplicated separators, missing spaces before "(", unclosed parens,
          empty brackets, and multi-space
- artist: strips URL junk; normalises ft./featuring → feat.; collapses duplicate separators
- label:  clears pure placeholder junk (unknown/n/a/none/—/???); normalises whitespace
- Preview is the default; --apply required for writes
- Every change logged with a reason code to the text log
- Supports --limit N, --output-json FILE, --verbose

FILES:
- modules/metadata_sanitize.py  (new)
- pipeline.py  (subparser + dispatch added)

NOTES:
- Designed to slot first in the workflow:
    metadata-sanitize → ai-normalize → artist-intelligence → metadata-enrich-online
- Does not modify BPM, key, genre, or cue data
- Safe on clean libraries: zero changes if tags are already clean

[2026-04-18] — Fix version_sim scoring: both-absent case now scores 1.0
----------------------------------------------------------------------
TYPE: Bug fix
SUMMARY: When neither the current title nor the candidate title had a version/remix
         token, version_sim was 0.0 (penalising a correct match) instead of 1.0.
         Root cause: _similarity() short-circuits on empty strings, so
         _similarity("", "") returned 0.0 rather than indicating agreement.
DETAILS:
  Replaced the inline _similarity(_extract_version(a), _extract_version(b)) call
  in score_candidate() with a dedicated _version_sim(title_a, title_b) function
  that applies explicit rules:
    Both absent          → 1.0   ("Setter" ↔ "Setter" now scores correctly)
    Both present, same   → 1.0
    Both present, differ → 0.0   (hard mismatch, different recording)
    One present, one not → 0.0   (asymmetric; version unknown)

  Before fix — "Setter" ↔ "Setter" (perfect artist+title match, no version):
    version_sim = 0.0  →  composite = 0.35×1 + 0.30×1 + 0.25×0 + 0.10×0 = 0.65

  After fix:
    version_sim = 1.0  →  composite = 0.35×1 + 0.30×1 + 0.25×1 + 0.10×0 = 0.90

FILES:
  intelligence/enrichment/metadata_matcher.py
IMPACT: Tracks with no version token now score correctly (0.90 for a perfect
        artist+title match).  Mismatches and asymmetric cases unchanged.
        313 tests pass.

DETAILS:
  Weights (metadata_matcher.py):
    title_sim   0.45 → 0.30
    version_sim 0.10 → 0.25    (artist 0.35, label 0.10 unchanged)
    Weights sum to 1.00.

  Hard safeguards (metadata_matcher.best_match):
    Version mismatch: cap 0.50 → 0.74, clear changes (wrong recording).
    Artist guard:     cap at 0.74 when artist_sim < 0.90 (no ISRC anchor),
                      keep changes visible in preview.
    Neither fires for ISRC-confirmed matches.

  Label alias boost (metadata_matcher.score_candidate):
    _LABEL_ALIAS_GROUPS: 13 groups (Sony/Warner/Universal + Afro house labels).
    Boost: +0.05 capped at 1.0. Reflected in reason string as [label-alias+0.05].

    CSS selectors may need updating if Traxsource restructures its frontend.

  Source priority (runner._search_with_fallback):

  Schema changes (enrichment_schema.py):

    EnrichmentMatch.sources_tried — ordered list of sources queried
    EnrichmentMatch.match_reason  — property alias for .reason

    Terminal output now shows "Sources tried: spotify → deezer → traxsource"

  too_low_10x    : 0 < BPM < 20  → suggest × 10  (e.g. 12.1 → 121)
  likely_halved  : 20 ≤ BPM < 90 → suggest × 2   (e.g. 64 → 128)

[2026-04-12] v1.7.0 — Artist merge: three-category classification, normalization fix, tests

TYPE: feature / bug fix

SUMMARY:
Refactored artist-merge classification so that punctuation/casing/hyphen variants
(e.g. "Heavy-K" / "Heavy K", "H.O.S.H" / "HOSH") are correctly recognised as
safe aliases instead of being marked ambiguous. Introduced three explicit merge
categories and 77 new tests.

DETAILS:
Root cause of the bug:
  The old _classify_merge() compared fi.primary_artist.lower() directly.
  "Heavy-K".lower() = "heavy-k" ≠ "heavy k" = "Heavy K".lower(), so despite
  being grouped correctly, the comparison flagged them as different primaries.

Fix:
  New _normalize_primary_for_compare() applies the same transforms used by
  normalize_artist_key() (period strip, hyphen→space, etc.) before comparison.
  Now "Heavy-K" and "Heavy K" both normalize to "heavy k" → SAFE_ALIAS.

Three merge categories (replacing the old binary safe/unsafe):
  SAFE_ALIAS          — formatting/punctuation/casing differences only; no collab.
                        Examples: "Heavy-K" / "Heavy K", "H.O.S.H" / "HOSH",
                        "Rosalie" / "Rosalie.", "culoe de song" / "Culoe De Song"
  SAME_PRIMARY_COLLAB — same primary after normalization; ≥1 folder has feat/comma.
                        Examples: "Heavy-K" / "Heavy-K feat. Davido"
  AMBIGUOUS           — primary artists differ after normalization; never auto-merged.

  - _describe_alias_differences() generates human-readable reason strings:
      "capitalization variant", "dotted-initials variant", "hyphen/space variant",
      "trailing period variant", "period variant", "underscore/space variant",
      "quotation style variant"
  - _pick_canonical() penalises all-uppercase names (e.g. HOSH loses to H.O.S.H)
    and all-lowercase names; mixed-case preferred.
  - FolderInfo.has_collab_suffix field added (populated at scan time).
  - MergeGroup.merge_category field added.
  - build_merge_groups() returns (safe, ambiguous) where safe = SAFE_ALIAS +
    SAME_PRIMARY_COLLAB groups; ambiguous = AMBIGUOUS only.
  - JSON report restructured with safe_alias_merges, same_primary_collab_merges,
    ambiguous_merges sections.
  - Console output shows all three categories with clear labels.
  - New constants: MERGE_CATEGORY_SAFE_ALIAS, MERGE_CATEGORY_SAME_PRIMARY_COLLAB,
    MERGE_CATEGORY_AMBIGUOUS.

SUMMARY:

classifies every audio file into LOSSLESS/HIGH/MEDIUM/LOW/UNKNOWN tiers, and
generates CSV/JSON reports.  Optional actions (both off by default): move LOW

DETAILS:
Quality tier rules:
  LOSSLESS  FLAC / ALAC / WAV / AIFF (codec-based; bitrate irrelevant)

  - probes files with ffprobe
  - classifies each file
  - writes CSV + JSON report to logs/reports/audit_quality/
  - prints terminal summary with counts + percentages per tier
  - no file moves, no tag writes

  --write-tags            write QUALITY tag (MP3: TXXX:QUALITY, FLAC: QUALITY,

  --dry-run               log intended actions; make no changes; skip report files
  --min-lossy-kbps N      shift LOW/MEDIUM boundary (default 192; HIGH fixed at 256)
  --report-format csv,json  control output formats (default: both)

[2026-04-11] v1.3.0 — Sanitizer: strip leading track-index junk before pipe
----------------------------------------------------------------------

TYPE: feature / patch

SUMMARY:
- New regex rule in sanitize_text() removes junk prefixes (digit, single letter,
  or symbol+digit/letter) that appear at the very start of a title before a `|`
  character and a space.  Handles DJ-pool numbering like "2 | Heartbreaker" or
  "***3 | Track (Remix)" and single-letter prefixes like "A | Charlotte".

DETAILS:
- Pattern: `^[^\w\s]*(?:\d+|[A-Za-z])\s*\|\s*`
  - `^`         — anchored to start; mid-string pipes are NEVER touched
  - `[^\w\s]*`  — zero or more leading symbol chars (e.g. `***`)
  - `(?:\d+|[A-Za-z])` — one or more digits OR exactly one letter (single-char only)
  - `\s*\|\s*`  — optional spaces around the pipe (all consumed)
- Multi-letter words before a pipe (e.g. "Rock | Roll", "Day | Night") are
  preserved because the single `[A-Za-z]` token requires the next char to be
  the pipe, not another word character.
- Runs as Step 0 in sanitize_text(), before URL and promo-phrase passes.
- 8 new test cases added to TestSanitizeText.

FILES:
- modules/sanitizer.py     — added _RE_PIPE_JUNK_PREFIX regex + Step 0 in sanitize_text()
- tests/test_sanitizer.py  — 8 new tests for the pipe-prefix rule

IMPACT:
- Any title/field string starting with a short numeric or single-letter index
  before a `|` will now be cleaned automatically in the sanitize pass.
- No impact on MIK BPM/key/cue data (sanitizer never touches those fields).

- This is additive; all 76 existing tests pass.
