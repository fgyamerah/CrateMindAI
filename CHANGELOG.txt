TYPE: Feature (intelligence/enrichment/runner.py, pipeline.py)
SUMMARY: Two operational features for the enrichment pipeline — persistent review queue
         for medium-confidence results, and automatic move-to-IGNORED for unresolvable files.

DETAILS:
  Review queue (enrichment_review_queue.json):
    _enqueue_for_review(): called automatically for every review/review_ambiguous result.

    Atomic save via .json.tmp rename.
    CLI: python3 pipeline.py review-queue [--list-only]

    review items are NOT moved (they go to the queue instead).
    Preserves folder structure: file.relative_to(music_root) → IGNORED/<rel>.
    Collision handling: appends _dup1, _dup2, … suffix.
    Off by default — only active with explicit --move-ignored flag.

  No changes to existing enrichment behavior without new flags.

[2026-04-20] — metadata-enrich-online: strict multi-factor validation layer
----------------------------------------------------------------------
TYPE: Hardening (intelligence/enrichment/)
SUMMARY: Replaced loose similarity-based matching with a strict gated pipeline.
         Goal: prefer skipping over incorrect matches; only apply when identity
         is highly confident.

DETAILS:
  Matching pipeline (gates applied in order before scoring):
    Gate A — Title similarity: strip mix/version tokens, require title_sim >= 0.92
    Gate B — Artist consistency: reject if artist_sim < 0.85 (no ISRC anchor)
    Gate C — Version alignment: reject conflicting version tokens (e.g. Remix vs Original)
  After gates, ambiguity check: if top-2 candidates within 0.05, mark review_ambiguous.

  Scoring formula changed to title-primary:
    OLD: artist(35%) + title(30%) + version(25%) + label(10%)
    NEW: title(50%) + artist(30%) + version(20%) + label as optional booster (+0.05)

  Decision thresholds raised:
    >= 0.90 → ready (was 0.80)
    0.75–0.89 → review (new band, needs human confirmation)
    < 0.75 → skip

  Machine-readable decision_code added to EnrichmentMatch:
    "ready" | "review" | "review_ambiguous"
    "skipped_low_score" | "skipped_artist_mismatch" | "skipped_version_conflict"

  Runner changes:
    - Loads artist alias store once before file loop; resolves canonical artist per file
    - Decision-code-driven apply logic (hard rejections never auto-apply even with --apply)
    - review and review_ambiguous cases counted separately (n_review)
    - JSONL logging includes decision_code on all records
    - Summary output shows Review queue count

FILES:
  intelligence/enrichment/metadata_matcher.py — complete rewrite
  intelligence/enrichment/enrichment_schema.py — added decision_code field
  intelligence/enrichment/runner.py — canonical artist, decision-code logic, review tracking
  config.py — ENRICH_ONLINE_MIN_CONFIDENCE: 0.80 → 0.90

IMPACT:
  CLI compatible (no flag changes). Stricter: more tracks will be skipped or queued
  for review vs the old system. ISRC exact matches (0.98) are unaffected.

NOTES:
  Duration booster deferred — requires adding duration_ms to EnrichmentCandidate
  and to Spotify/Deezer/Traxsource response parsers.

BPM LOGIC (classify_bpm):
  missing_bpm    : NULL or zero stored
  too_low_10x    : 0 < BPM < 20  → suggest × 10  (e.g. 12.1 → 121)
  likely_halved  : 20 ≤ BPM < 90 → suggest × 2   (e.g. 64 → 128)
  likely_doubled : 160 < BPM ≤ 240, not DNB/jungle/hardcore → suggest ÷ 2
  too_high       : BPM > 240 → suggest ÷ 2

WHY:
  Incorrect BPM (especially doubled/halved) is a common aubio artifact.
  The analyzer already corrects these at analysis time, but values that
  came from MIK, Rekordbox, or direct tag writes bypass that correction.
  This feature surfaces those cases without touching files.

  - Human review state (reviewed/ignored/requeued) is preserved across re-scans
  - reanalyze endpoint uses existing analyze-missing + --reanalyze via job system

[2026-04-12] v1.7.0 — Artist merge: three-category classification, normalization fix, tests
----------------------------------------------------------------------

TYPE: feature / bug fix

SUMMARY:
Refactored artist-merge classification so that punctuation/casing/hyphen variants
(e.g. "Heavy-K" / "Heavy K", "H.O.S.H" / "HOSH") are correctly recognised as
safe aliases instead of being marked ambiguous. Introduced three explicit merge
categories and 77 new tests.

DETAILS:
Root cause of the bug:
  The old _classify_merge() compared fi.primary_artist.lower() directly.
  "Heavy-K".lower() = "heavy-k" ≠ "heavy k" = "Heavy K".lower(), so despite
  being grouped correctly, the comparison flagged them as different primaries.

Fix:
  New _normalize_primary_for_compare() applies the same transforms used by
  normalize_artist_key() (period strip, hyphen→space, etc.) before comparison.
  Now "Heavy-K" and "Heavy K" both normalize to "heavy k" → SAFE_ALIAS.

Three merge categories (replacing the old binary safe/unsafe):
  SAFE_ALIAS          — formatting/punctuation/casing differences only; no collab.
                        Examples: "Heavy-K" / "Heavy K", "H.O.S.H" / "HOSH",
                        "Rosalie" / "Rosalie.", "culoe de song" / "Culoe De Song"
  SAME_PRIMARY_COLLAB — same primary after normalization; ≥1 folder has feat/comma.
                        Examples: "Heavy-K" / "Heavy-K feat. Davido"
  AMBIGUOUS           — primary artists differ after normalization; never auto-merged.

Other improvements:
  - _describe_alias_differences() generates human-readable reason strings:
      "capitalization variant", "dotted-initials variant", "hyphen/space variant",
      "trailing period variant", "period variant", "underscore/space variant",
      "quotation style variant"
  - _pick_canonical() penalises all-uppercase names (e.g. HOSH loses to H.O.S.H)
    and all-lowercase names; mixed-case preferred.
  - FolderInfo.has_collab_suffix field added (populated at scan time).
  - MergeGroup.merge_category field added.
  - build_merge_groups() returns (safe, ambiguous) where safe = SAFE_ALIAS +
    SAME_PRIMARY_COLLAB groups; ambiguous = AMBIGUOUS only.
  - JSON report restructured with safe_alias_merges, same_primary_collab_merges,
    ambiguous_merges sections.
  - Console output shows all three categories with clear labels.
  - New constants: MERGE_CATEGORY_SAFE_ALIAS, MERGE_CATEGORY_SAME_PRIMARY_COLLAB,
    MERGE_CATEGORY_AMBIGUOUS.

FILES:
  modules/artist_merge.py     — full refactor (normalization fix, new functions,
                                 updated dataclasses, report structure)
  tests/test_artist_merge.py  — new: 77 tests across 7 test classes

IMPACT:
  - Behaviour change: groups that were previously marked AMBIGUOUS (and skipped)
    because of hyphen/period/case differences are now correctly classified as
    SAFE_ALIAS and auto-applied in --apply mode.
  - No change to file-move logic, DB updates, or other pipeline steps.

[2026-04-11] v1.3.0 — Sanitizer: strip leading track-index junk before pipe
----------------------------------------------------------------------

TYPE: feature / patch

SUMMARY:
- New regex rule in sanitize_text() removes junk prefixes (digit, single letter,
  or symbol+digit/letter) that appear at the very start of a title before a `|`
  character and a space.  Handles DJ-pool numbering like "2 | Heartbreaker" or
  "***3 | Track (Remix)" and single-letter prefixes like "A | Charlotte".

DETAILS:
- Pattern: `^[^\w\s]*(?:\d+|[A-Za-z])\s*\|\s*`
  - `^`         — anchored to start; mid-string pipes are NEVER touched
  - `[^\w\s]*`  — zero or more leading symbol chars (e.g. `***`)
  - `(?:\d+|[A-Za-z])` — one or more digits OR exactly one letter (single-char only)
  - `\s*\|\s*`  — optional spaces around the pipe (all consumed)
- Multi-letter words before a pipe (e.g. "Rock | Roll", "Day | Night") are
  preserved because the single `[A-Za-z]` token requires the next char to be
  the pipe, not another word character.
- Runs as Step 0 in sanitize_text(), before URL and promo-phrase passes.
- 8 new test cases added to TestSanitizeText.

FILES:
- modules/sanitizer.py     — added _RE_PIPE_JUNK_PREFIX regex + Step 0 in sanitize_text()
- tests/test_sanitizer.py  — 8 new tests for the pipe-prefix rule

IMPACT:
- Any title/field string starting with a short numeric or single-letter index
  before a `|` will now be cleaned automatically in the sanitize pass.
- No impact on MIK BPM/key/cue data (sanitizer never touches those fields).
